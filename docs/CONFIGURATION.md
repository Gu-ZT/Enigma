# Go Configuration Guide

[中文说明](./CONFIGURATION.zh_CN.md)

ETP/1 is configured with `enigma.Config` when an existing `net.Conn` is
wrapped. There are no JSON files or client/server roles in the current package:
both peers use the same API, and each connection direction initializes itself
independently.

## Minimal Configuration

```go
cfg := enigma.Config{
	Key: sharedKey, // at least 32 random bytes
}

conn, err := enigma.NewConn(rawConn, cfg)
if err != nil {
	return err
}
```

With this configuration, record padding and cover padding are disabled, the
default printable alphabet is used, and records carry at most 16 KiB of payload.

## Balanced Example

```go
cfg := enigma.Config{
	Key: sharedKey,

	MinPadding: 4,
	MaxPadding: 64,

	MinCoverPadding: 2,
	MaxCoverPadding: 32,

	MaxPayload: 16 * 1024,
}
```

Padding values are byte/character counts per record, not percentages. Larger
ranges increase wire overhead and do not hide timing or total connection size.

## Fields

### Key

`Key` is required and must contain at least 32 bytes. It should be generated by
`crypto/rand` and distributed through an authenticated secure channel. ETP/1
does not turn a password into a strong key and does not negotiate keys.

```go
key := make([]byte, 32)
if _, err := rand.Read(key); err != nil {
	return err
}
```

### CoverAlphabet

`CoverAlphabet` must contain exactly 64 unique printable ASCII bytes in the
range `0x21` through `0x7e`.

Default:

```text
ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789-_
```

The alphabet order is part of the wire profile. Peers must use the same value.
Changing it does not strengthen encryption; it only changes printable symbols.

### PaddingAlphabet

`PaddingAlphabet` lists bytes that the decoder may ignore between encoded
symbols. Each byte must be printable ASCII or tab/CR/LF and must not occur in
`CoverAlphabet`. Duplicate padding bytes are accepted and normalized.

If the field is empty, ETP/1 uses its default whitespace-and-punctuation set.
Unexpected bytes outside both alphabets fail with `ErrUnexpectedCoverByte`.

### MinPadding and MaxPadding

These values bound random bytes appended inside the authenticated plaintext of
every record.

- default: `0`, `0`;
- valid range: `0` through `8192`;
- `MinPadding` must not exceed `MaxPadding`.

The receiver authenticates these bytes before discarding them. A receiver's
`MaxPadding` must be at least as large as the sender's possible record padding.

### MinCoverPadding and MaxCoverPadding

These values bound ignorable padding characters inserted into each encoded
session salt or frame.

- default: `0`, `0`;
- valid range: `0` through `8192`;
- `MinCoverPadding` must not exceed `MaxCoverPadding`.

Cover padding is filtered before rotor decoding and is not included in the AEAD
record. Only configured padding characters are ignored.

### MaxPayload

`MaxPayload` is the maximum payload stored in one authenticated record.

- default: `16384` bytes;
- valid range: `1` through `32768` bytes;
- larger calls to `Write` are split into multiple records.

The reader uses its own `MaxPayload` as a hard allocation and validation limit.
Peers should configure the same value.

## Interoperability Requirements

| Setting | Must match? | Reason |
| --- | --- | --- |
| `Key` | Yes | Derives AEAD keys, rotors, masks, and nonces |
| `CoverAlphabet` | Yes | Defines symbol-to-index decoding |
| `PaddingAlphabet` | Yes | Defines which non-cover bytes may be ignored |
| `MaxPayload` | Recommended | Receiver rejects larger records |
| `MaxPadding` | Receiver must allow sender | Receiver rejects excessive inner padding |
| Minimum padding values | No | They only control local sender choices |
| Cover padding ranges | No | Decoder filters by alphabet, not count |

Using the same complete `Config` at both ends is the simplest interoperable
setup.

## Validation

`enigma.NewConn` validates configuration before performing network I/O. It
rejects short keys, invalid alphabets, overlapping cover/padding bytes, reversed
padding ranges, and out-of-bounds limits.

Configuration errors are regular contextual errors. Wire errors can additionally
be classified with `errors.Is` against `ErrAuthentication`, `ErrInvalidFrame`,
and `ErrUnexpectedCoverByte`.
